# ConstructionFront Project Intelligence Database Workflow

This workflow turns ConstructionFront article titles and URLs into internal project intelligence records. It is intentionally conservative: title-derived facts stay in review until a human checks the article and supporting source material.

## Folder Layout

- `data/project-intelligence/projects_raw.csv` stores the imported v0 source records. Do not edit this file directly.
- `data/project-intelligence/projects_clean.csv` stores normalized records in the approved database field order.
- `data/project-intelligence/projects_review_queue.csv` lists records that need human review and the reason.
- `data/project-intelligence/project_change_log.csv` records normalization changes and generated draft records.
- `data/project-intelligence/taxonomies.json` contains allowed values for controlled fields.
- `scripts/project-intelligence/normalize-project-records.ts` converts raw records into clean records and review queue rows.
- `scripts/project-intelligence/validate-projects.ts` validates clean records against taxonomies and business rules.
- `scripts/project-intelligence/create-project-record-from-article.ts` creates a draft record from a new article title and URL.

## Add A New News Article

Create a draft record from the article title and URL:

```powershell
node scripts/project-intelligence/create-project-record-from-article.ts --title "Article title" --url "https://constructionfront.com/article-url/" --date 2026-05-21 --append
```

The script can infer a rough sector or stage from article-title keywords, but it does not approve the record. Unknown owners, contractors, values, dates, and coordinates remain blank or marked for review.

## Generate A Draft Project Record

Without `--append`, the script prints the draft record as JSON so it can be inspected before adding it to the database:

```powershell
node scripts/project-intelligence/create-project-record-from-article.ts --title "Article title" --url "https://constructionfront.com/article-url/"
```

Use this mode when the article may describe an update to an existing project rather than a new project. Check `projects_clean.csv` for a matching slug, article URL, country, and location before appending.

## Normalize The Database

Run normalization after importing or editing `projects_raw.csv`:

```powershell
node scripts/project-intelligence/normalize-project-records.ts
```

Normalization maps the v0 workbook fields into the internal schema, converts non-core regions into `Other`, converts title-derived confidence into `Needs Review`, normalizes map readiness to boolean text, and rebuilds the review queue and change log.

## Validate The Database

Run validation before approving records or using them in downstream products:

```powershell
node scripts/project-intelligence/validate-projects.ts
```

Validation checks required fields, unique slugs, approved taxonomy values, coordinate requirements for map readiness, BD timing consistency, source-review triggers, and duplicate or near-duplicate signals.

## Approve Or Reject Records

Review `projects_review_queue.csv` row by row. For each record:

1. Open the ConstructionFront article and any official source links.
2. Confirm or correct project name, location, sector, project stage, procurement status, FID status, owners, contractors, values, and dates.
3. Add coordinates only when the location is supported by a source.
4. Update `confidenceLevel` to `Confirmed`, `Likely`, or `Inferred` only when the evidence supports it.
5. Set `reviewStatus` to `Approved` only after required checks are complete.
6. Set `Rejected` only for records that are duplicates, irrelevant, or unsuitable for the database.
7. Add every material correction to `project_change_log.csv`.

Never silently approve uncertain data. Use `Unknown`, blank fields, or a `Needs ...` review status when source evidence is incomplete.

## Future Project Map Connection

The public map should only read approved records from `projects_clean.csv`. A record should be map-eligible only when:

- `reviewStatus` is `Approved`
- `mapReady` is `true`
- `latitude` and `longitude` are present
- `includeInV0Map` is `true`
- the project is inside core coverage or has been deliberately approved as extended coverage

The map should not write directly to the intelligence database. Future map tooling should consume a reviewed export generated from this internal workflow.
