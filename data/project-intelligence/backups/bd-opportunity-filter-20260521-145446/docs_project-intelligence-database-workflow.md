# ConstructionFront Project Intelligence Database Workflow

This workflow turns ConstructionFront article titles and URLs into internal project intelligence records. It is intentionally conservative: title-derived facts stay in review until a human checks the article and supporting source material.

Current schema version: v0.2. See `docs/project-intelligence-schema-v0-1.md` for project master field definitions and `docs/project-intelligence-schema-v0-2.md` for the simplified update/event `milestoneType` taxonomy.

## Folder Layout

- `data/project-intelligence/projects_raw.csv` stores the imported v0 source records. Do not edit this file directly.
- `data/project-intelligence/projects_clean.csv` stores the original v0 normalized records and should be preserved.
- `data/project-intelligence/projects_review_queue.csv` stores the original v0 review queue and should be preserved.
- `data/project-intelligence/project_change_log.csv` stores the original v0 normalization log and should be preserved.
- `data/project-intelligence/projects_clean_v0_1.csv` stores normalized records in the current v0.1 schema.
- `data/project-intelligence/projects_review_queue_v0_1.csv` lists v0.1 records that need human review and the reason.
- `data/project-intelligence/project_change_log_v0_1.csv` records v0.1 schema migration, normalization changes and generated draft records.
- `data/project-intelligence/sample/projects_master_sample_5_clean.csv` stores the v0.2 clean 5-project master-table sample.
- `data/project-intelligence/sample/project_updates_sample_5_clean.csv` stores the v0.2 clean update/event sample with controlled `milestoneType` and `milestoneSummary`.
- `data/project-intelligence/taxonomies.json` contains allowed values for controlled fields.
- `scripts/project-intelligence/normalize-project-records.ts` converts raw records into clean records and review queue rows.
- `scripts/project-intelligence/validate-projects.ts` validates clean records against taxonomies and business rules.
- `scripts/project-intelligence/create-project-record-from-article.ts` creates a draft record from a new article title and URL.

## Add A New News Article

Create a draft record from the article title and URL:

```powershell
node scripts/project-intelligence/create-project-record-from-article.ts --title "Article title" --url "https://constructionfront.com/article-url/" --date 2026-05-21 --append
```

The script can infer a rough sector, lifecycle stage, pre-construction approach and opportunity categories from article-title keywords, but it does not approve the record. Unknown owners, contractors, values, dates, quarters and coordinates remain blank or marked for review.

## Generate A Draft Project Record

Without `--append`, the script prints the draft record as JSON so it can be inspected before adding it to the database:

```powershell
node scripts/project-intelligence/create-project-record-from-article.ts --title "Article title" --url "https://constructionfront.com/article-url/"
```

Use this mode when the article may describe an update to an existing project rather than a new project. Check `projects_clean_v0_1.csv` for a matching slug, article URL, country, and location before appending.

## Normalize The Database

Run normalization after importing or editing `projects_raw.csv`:

```powershell
node scripts/project-intelligence/normalize-project-records.ts
```

Normalization maps the v0 workbook fields into the v0.1 internal schema, converts non-core regions into `Other`, maps the former `FEED / Design Development` stage to `Pre-Construction / Design`, adds `preConstructionApproach`, adds structured schedule fields, copies `commercialOpportunity` into `potentialOpportunities`, adds controlled `opportunityCategories`, converts title-derived confidence into `Needs Review`, normalizes map readiness to boolean text, and rebuilds the v0.1 review queue and change log.

## Validate The Database

Run validation before approving records or using them in downstream products:

```powershell
node scripts/project-intelligence/validate-projects.ts
```

Validation checks required fields, unique slugs, approved taxonomy values, coordinate requirements for map readiness, BD timing consistency, pre-construction approach consistency, structured schedule precision, opportunity categories, source-review triggers, and duplicate or near-duplicate signals.

For project update/event tables, validation checks `updateId`, `projectSlug`, article/source fields, controlled `milestoneType`, and `milestoneSummary` when present.

`currentProjectStage` and `milestoneType` must stay separate: the former is the current lifecycle state of the project; the latter is what the article/update is about. Put specific details such as practical completion, testing, opened to traffic, full notice to proceed, or construction start in `milestoneSummary`.

For broad national programmes, country-level portfolios, financing records or grid expansion programmes without a discrete physical site, records may use a country-capital display anchor. These rows must use `locationPrecision = National`, `mapAnchorType = Country Capital Anchor` or `Programme / Portfolio Anchor`, and `mapEligibilityStatus = Map Ready - Approximate`. The `mapEligibilityNotes` field must clearly state that the coordinates are a display anchor only and not the physical project location.

Financial figures should be stored as display-scale values. Use `projectValueAmount`, `projectValueCurrency`, `projectValueScale`, `projectValueBasis` and `projectValueNotes` on the project master table. Use `eventValueAmount`, `eventValueCurrency`, `eventValueScale`, `eventValueBasis` and `eventValueNotes` on update/event rows when an article is about a contract award, financing close, debt facility, PPA/offtake, settlement, phase value or similar event. Do not store raw values such as `177000000` when `177 / EUR / Million` is the intended editorial value.

## Approve Or Reject Records

Review `projects_review_queue.csv` row by row. For each record:

1. Open the ConstructionFront article and any official source links.
2. Confirm or correct project name, location, sector, project stage, procurement status, FID status, owners, contractors, values, and dates.
3. Add coordinates only when the location is supported by a source.
4. Set `preConstructionApproach` only where the source or project pathway supports the specific approach. Do not merge FEED/FEL and ECI.
5. Use structured schedule fields. Do not infer quarters. Use `Indicative` precision for expected, planned, targeted or anticipated dates.
6. Populate `potentialOpportunities` and `opportunityCategories` only where they are supported by the project type and stage.
7. Update `confidenceLevel` to `Confirmed`, `Likely`, or `Inferred` only when the evidence supports it.
8. Set `reviewStatus` to `Approved` only after required checks are complete.
9. Set `Rejected` only for records that are duplicates, irrelevant, or unsuitable for the database.
10. Add every material correction to `project_change_log_v0_1.csv`.

Never silently approve uncertain data. Use `Unknown`, blank fields, or a `Needs ...` review status when source evidence is incomplete.

## Future Project Map Connection

The public map should only read approved records from the current reviewed export, not directly from draft workflow files. For v0.1, a record should be map-eligible only when:

- `reviewStatus` is `Approved`
- `mapReady` is `true`
- `latitude` and `longitude` are present
- `includeInV0Map` is `true`
- the project is inside core coverage or has been deliberately approved as extended coverage

The map should not write directly to the intelligence database. Future map tooling should consume a reviewed export generated from this internal workflow.
